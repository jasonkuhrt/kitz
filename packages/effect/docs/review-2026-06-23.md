# @kitz/effect — Review Findings (2026-06-23)

Source: a 6-agent parallel audit of `@kitz/effect` (FileSystem service, Path ADT,
path analyzer/types, builder, packaging, memory layer). Findings are hard and
source-backed; the auditing agents verified most empirically, and the four
highest-severity data-integrity claims were independently re-confirmed by the
orchestrator (marked **✔confirmed**).

**Status:** ⬜ open · ✅ resolved · 🔶 partial / accepted gap · 🧭 contract
decision (needs an owner call before a fix)

---

  (`RemoveTrailingSlash` now strips all), `ANL-L3` (`AnalysisFile.file` trailing-slash

## D. Builder — `builder/*`

Root cause for D-H1/H3/H4: inputs are typed as if extensions disambiguate
file/dir, but `Path.fromString` disambiguates by **trailing slash only** for raw
strings, and content is erased to `any` at the `Operation` boundary — so the
compiler validates the call sites but nothing validates the spec→commit data flow.

| ID | Sev | Location | Issue | Fix |
|---|---|---|---|---|
| BLD-H1 | HIGH ✔confirmed | `spec.ts:452,372` → `operations.ts:35` → `filesystem.ts:672` | `add('src')` (no slash) becomes a *file*; committing writes the literal string `"undefined"` as content | Trust the brand for typed `Path.Rel*`; require explicit nature (or trailing slash) for raw strings |
| BLD-H2 | HIGH | (whole `builder/`) | Zero tests for a 851-line public subsystem | `spec.test.ts` covering disambiguation, `toLayout`, move/merge/when |
| BLD-H3 | HIGH | `spec.ts:18` | `Operation.file.content: any` erases all content typing past the call site | Type as `string \| Uint8Array \| JsonObject` |
| BLD-H4 | HIGH | `spec.ts:376` | `remove` ternary is a provable no-op (both branches return `fsPath`) | `const operationPath = fsPath` |
| BLD-M1 | MED | `spec.ts:394-450` | `move()` throws on type-valid mixed-slash input; `move-file`/`move-dir` both just `rename` | Collapse to one `move` op |
| BLD-M2 | MED | `chain.ts:21-71` | ~50 lines of overload duplication between `SpecBuilder` and `DirChain` | Generic self-type interface |
| BLD-M3 | MED | `chain.ts:99,127` | Dead mutable `let spec` (write never read) | Delete |
| BLD-M4 | MED | `operations.ts:39-50` | `dir` op redundantly `makeDirectory`s (files already create parents) | Only create empty dirs; else recurse |
| BLD-M5 | MED | `chain.ts:71`, `spec.ts:532-538` | `merge()` discards operands' base dirs → wrong absolute paths | Assert equal bases or rebase |
| BLD-M6 | MED | `chain.ts:148-150` | `withChaining` relies on `Object.assign` field coincidence | `return chain(builder)` |
| BLD-L1 | LOW | `spec.ts:385` | `clear()` special-cases `'.'`→`'./'`; no other method does | Normalize uniformly |
| BLD-L2 | LOW | `spec.ts:206-214,250-254` | Stale `withBase` JSDoc + dead `// TODO: import feature` in public interface | Fix doc; move TODO out |
| BLD-L3 | LOW | `spec.ts:394,452` | `add`/`move` impl params are `any` (why BLD-H1 escaped the compiler) | Type as `Path.$Rel \| string` |
| BLD-L4 | LOW | `spec.ts:11,289` | `Layout` value type diverges from what `toLayout` stores | Stringify/reject non-string/Uint8Array for non-`.json` |
| BLD-X1 | MED | `filesystem.ts:34` | `InferContentFromExtension` maps `.json`→`JsonObject` (too narrow — rejects top-level JSON arrays) | Use `JsonValue` (already exists) |

Note: `InferContentFromExtension`/`InferFileContent` themselves are **sound** (no
`any` hole) — the only defects are BLD-X1 and the `any` erasure at the data
boundary (BLD-H3). `spec.ts`'s only cleanly-extractable seam is `toLayout`
(`:257-304`); the rest is cohesive — its debt is correctness, not size.
